## high: Pytest suite contains no executable assertions

id: fnd_sig-feat-test-suite-a6226f638f-1_f236061f2e
category: test-gap
confidence: high
triage: test-gap
status: open
feature: Python test suite tests (feat_test-suite_a6226f638f)

evidence:
- smoke_test.py:19-20
- test_api.py:1-2
- test_api_v2.py:1-2
- test_api.py:27-28
- test_api_v2.py:27-28

The owned files are presented as pytest tests, but none define test functions, fixtures, or assertions. smoke_test.py only runs its checks under __main__, which pytest will not execute during normal collection. test_api.py and test_api_v2.py run side-effect code at import time and swallow top-level failures by printing them, so API or database failures are not represented as test failures. This means the claimed test-suite feature does not reliably validate the application behavior it appears to cover.

recommendation:
Convert these scripts into real pytest tests: move side-effect code into functions named test_*, use fixtures for API base URL and database connection, and replace print-only checks with assertions on HTTP status codes, JSON shape, and expected database/API relationships. Do not catch broad exceptions unless the test re-raises or explicitly fails.

test analysis:
These files are the tests under review; their current structure is the reason failures are not enforced.

suggested regression test:
Add a pytest test that monkeypatches or fixtures the API/DB dependencies, forces an endpoint failure, and asserts pytest reports a failing test rather than only printing the error.

minimum fix scope:
Refactor smoke_test.py, test_api.py, and test_api_v2.py into collected pytest test functions with assertions and failing exception behavior.

repro:
Run pytest against these files. There are no collected test items/assertions, and failures inside test_api.py/test_api_v2.py are caught and printed instead of failing the run.

## medium: Extracted names can include all following OCR fields

id: fnd_sig-feat-library-25faef2bc4-2f0b_9dd13b0590
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Python source root (feat_library_25faef2bc4)

evidence:
- ocr_test_run.py:44-45 (extract_text)
- ocr_test_run.py:65-69 (parse_nid_fields)

`extract_text` flattens OCR output into one space-separated string with no newlines, but `parse_nid_fields` stops the name capture only at a newline. For typical OCR output containing `Name: Alice DOB: ... NID: ...`, the extracted name becomes `Alice DOB: ... NID: ...` rather than just the name.

recommendation:
Preserve OCR line boundaries when possible, or stop the name capture at known following labels such as DOB, Date of Birth, NID, ID No, or Bangla equivalents. Add parsing tests for text where fields appear on one OCR line.

test analysis:
No linked tests exercise `parse_nid_fields`, including the common case where EasyOCR returns field labels as adjacent fragments that are later joined with spaces.

suggested regression test:
Assert that `parse_nid_fields('Name: Alice Date of Birth: 01/01/1990 NID 1234567890')` returns `name == 'Alice'` while still extracting dob and nid.

minimum fix scope:
Update `extract_text` or the name regexes in `parse_nid_fields` and add focused parser tests.

repro:
Call `parse_nid_fields('Name: Alice Date of Birth: 01/01/1990 1234567890')`; the returned `name` includes the date and NID text after Alice.

repro:
Run `python ocr_test_run.py high_resolution_image.jpg` with an image whose largest side is greater than 1600 pixels; the original file is replaced by a smaller image before OCR runs.

## medium: Smoke test mutates live application state without isolation or verification

id: fnd_sig-feat-test-suite-a6226f638f-1_9c2925906e
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Python test suite tests (feat_test-suite_a6226f638f)

evidence:
- smoke_test.py:6-7
- smoke_test.py:35-45
- smoke_test.py:49-52

The smoke test selects the first unassigned police request from the running service and posts an assignment to a real localhost endpoint, including a fallback that assigns the case to the first available officer even when the area does not match. There is no fixture data, cleanup, transaction rollback, or assertion that the target service is a disposable test instance. Running this against a developer or shared database can permanently change case assignments and make subsequent tests or manual workflows observe corrupted state.

recommendation:
Make smoke tests operate only on isolated test data. Create a request and officer inside the test setup, assign only that fixture request, assert the response and resulting state, and clean up afterward. Add an explicit TESTING/SMOKE_ALLOW_MUTATION guard or require a dedicated test database/base URL before performing POST requests.

test analysis:
The smoke test performs the mutation itself and only prints the response; it has no cleanup or assertions that would detect accidental mutation of non-test data.

suggested regression test:
Add a pytest smoke test using a dedicated test database or mocked API where the test creates its own unassigned request, assigns it, verifies the assignment, and tears down the created records.

minimum fix scope:
Restrict smoke_test.py mutations to isolated fixture records and add setup, assertions, and cleanup around /assign-case.

repro:
Start the application against a database containing at least one unassigned request and one officer, then run smoke_test.py directly. The script posts to /assign-case for the first request it finds and does not undo the assignment.

## low: Template script creates project files as an import side effect

category: build-release
confidence: high
triage: risk
status: open
feature: Python source root (feat_library_25faef2bc4)

evidence:
- template.py:42-54

All filesystem creation runs at module top level. Any test discovery, import, static analysis helper, or packaging step that imports `template.py` will create directories and empty files in the current working directory, which can pollute builds and developer worktrees.

recommendation:
Move the creation logic into a `main()` function and invoke it only under `if __name__ == '__main__':`. Consider requiring an explicit target directory argument.

test analysis:
No linked tests import `template.py` from an isolated working directory and assert that import is side-effect free.

suggested regression test:
Import `template.py` in a temporary directory and assert no project files are created until `main()` is called explicitly.

minimum fix scope:
Wrap the folder/file creation loops in `main()` and add a `__main__` guard.

repro:
From a clean temporary directory with this file on `PYTHONPATH`, run `python -c "import template"`; it creates `backend`, `frontend`, `requirements.txt`, and `README.md` in the process working directory.
